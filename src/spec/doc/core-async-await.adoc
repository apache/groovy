= Async/Await
:jdk: https://docs.oracle.com/en/java/javase/21/docs/api

[[async-await]]
== Introduction

Groovy provides native `async`/`await` support as a language-level feature, enabling developers to write
asynchronous code in a sequential, readable style. Inspired by similar constructs in JavaScript, C#, Kotlin,
Swift, and Go,
Groovy's async/await integrates seamlessly with the JVM concurrency model while maintaining Groovy's hallmark
conciseness and expressiveness.

Key capabilities include:

* **`async` methods** â€” declare methods that execute asynchronously and return an `Awaitable`
* **`await` expressions** â€” suspend execution until an asynchronous result is available; supports single-expression (`await task`), parenthesized multi-arg (`await(a, b, c)`), and unparenthesized multi-arg (`await a, b, c`) forms for parallel awaiting
* **Async closures and lambdas** â€” create reusable asynchronous functions
* **`for await`** â€” iterate over asynchronous data sources
* **`yield return`** â€” produce asynchronous streams (generators)
* **`defer`** â€” schedule cleanup actions that run when the method completes (LIFO order)
* **`AsyncContext`** â€” propagate request, trace, and tenant metadata across async boundaries (interface with factory methods, `getOrDefault`, and `computeIfAbsent`)
* **`AsyncScope`** â€” structured concurrency scopes (interface with factory methods and Supplier-based Java interop)
* **`Awaitable.go`, `AsyncChannel`, and `Awaitable.any`** â€” task spawning and channel-based communication primitives
* **Framework integration** â€” built-in adapters for `CompletableFuture` and `Future`;
`Flow.Publisher` support via an auto-discovered runtime adapter;
drop-in Reactor and RxJava 3 modules (`groovy-reactor`, `groovy-rxjava`) with
`ServiceLoader`-based auto-discovery; extensible to any framework via the adapter registry

On JDK 21+, async methods automatically leverage
{jdk}/java.base/java/lang/Thread.html#ofVirtual()[virtual threads] for optimal scalability.

[[why-async-await]]
== Why Groovy Needs Async/Await

Modern applications are overwhelmingly concurrent. Web servers, microservices, data
pipelines, and interactive UIs all spend most of their time waiting â€” waiting for
network responses, database queries, file I/O, or downstream services. The question
is not _whether_ to write asynchronous code, but _how pleasant the experience will
be_.

=== Virtual Threads: Free Scalability

On JDK 21+, every `async` method runs on a
{jdk}/java.base/java/lang/Thread.html#ofVirtual()[virtual thread].
When the thread blocks on `await`, the JVM automatically parks the virtual thread
and releases the underlying carrier (OS) thread. This means millions of concurrent
`async` tasks can coexist on a handful of OS threads â€” achieving the same practical
scalability as compiler-generated state machines in C# or Kotlin, but with
_dramatically simpler_ compiler implementation and full debugging support
(complete stack traces, breakpoints, step-through debugging all work unmodified).

On JDK 17â€“20, Groovy falls back to a managed executor while preserving the
same user-facing async programming model.

=== Why a Language Feature, Not a Library?

GPars, RxJava, and Reactor already provide powerful async toolkits for the JVM. So
why build async/await into the _language_?

[cols="1,3"]
|===
| Reason | Explanation

| **Readability**
| `await` makes asynchronous code indistinguishable from synchronous code.
Libraries can provide combinators, but they cannot change the way `for`, `try`,
`if`, and assignment statements compose.

| **Exception transparency**
| Language-level `await` can intercept and unwrap JVM exception wrappers before they
reach the programmer. A library-level `await()` method cannot prevent `CompletionException`
from leaking into user code without bytecode manipulation.

| **Control-flow integration**
| `for await` integrates with the parser and AST. There is no way for a library to add
new loop syntax. Similarly, `yield return` requires the compiler to rewrite the method
into a generator â€” a transformation that no library can perform.

| **`defer` semantics**
| `defer` requires compiler-generated `try`/`finally` wrapping with LIFO
execution ordering. This cannot be emulated with a simple method call.

| **Tooling**
| Syntax highlighting, code completion, and static analysis can recognize `async`,
`await`, `for await`, and `yield return` as first-class constructs. Library-level
equivalents are just method calls to the IDE.
|===

In short, async/await elevates asynchronous programming from a _library concern_
to a _language concern_, giving Groovy the same expressive power that developers
have come to expect in JavaScript, C#, Kotlin, and Swift.

=== Why Keywords, Not Global Methods

The previous section explains why async/await belongs in the _language_ rather than
in a library. A more subtle question follows: even within the language, why introduce
reserved keywords instead of predefined global methods named `async`, `await`, and
`defer`? Groovy already provides built-in methods such as `println` â€” could
`await(expr)` work the same way?

The fundamental reason is that these constructs introduce _new syntactic forms_ that
the parser must recognize, and trigger _compile-time code transformations_ that must
complete before the program runs â€” requirements that go beyond what any method call
can express.

**New syntactic forms that methods cannot express.**
Several async constructs occupy syntactic positions where method calls are
grammatically impossible:

* `async` serves as a _method modifier_, appearing in the same syntactic position as
  `static` or `final` â€” for example, `async def fetch()` or `async String lookup()`.
  Modifiers are a distinct grammar category that precedes the return type and method
  name; a method call cannot fill this role.
* `for await (item in source)` embeds the `await` keyword into the loop header.
  There is no way for a method call to extend the `for` statement syntax.
* `yield return expr` is a two-keyword compound statement that signals the compiler
  to transform the enclosing method into an async generator. Two-word method names
  are not expressible in standard call syntax.
* `await expr` â€” without parentheses â€” is a dedicated grammar production that can
  appear in any expression context: assignments, method arguments, return statements,
  and complex expressions. The grammar also supports `await a, b, c` as a
  comma-separated multi-arg form, which desugars to
  `await Awaitable.all(a, b, c)`.  A parenthesis-free method call would follow
  Groovy's command expression rules, which have contextual restrictions and can
  interact unpredictably with surrounding syntax.
* `async { ... }` must be distinguished from a method call named `async` with a
  trailing closure argument. The grammar resolves this by placing the `async`
  closure/lambda rule before general method-call rules, so the parser treats `async`
  as a keyword rather than a method name. Without keyword status, `async { ... }`
  would be parsed as an ordinary method invocation, preventing the compiler from
  applying the async transformation.

These are not edge cases; they are the _primary_ syntax of the feature. If `async`,
`await`, and `defer` were methods, the most natural forms of async code could not be
written.

**Compile-time context validation.**
The compiler maintains an _async context depth_ counter during parsing. When it
encounters `await`, `defer`, or `yield return` outside an `async` method or closure,
it reports a compile-time error immediately â€” before the program runs. A global
method can only detect misuse at runtime, turning what should be an instant
compilation error into a latent `IllegalStateException` that may surface only under
specific execution paths.

**AST rewriting and control-flow transformation.**
Each keyword triggers a distinct structural transformation of the abstract syntax tree:

* `async` extracts the entire method body into a closure, captures parameters as
  closure-shared variables, and delegates execution to the async runtime.
* `await` with multiple comma-separated arguments â€” whether parenthesized
  (`await(a, b, c)`) or unparenthesized (`await a, b, c`) â€” is rewritten to
  `AsyncSupport.await(Awaitable.all(a, b, c))`, enabling parallel awaiting as
  a single expression.
* `defer` injects a synthetic scope variable and wraps the method body in a
  compiler-generated `try`/`finally` block to guarantee LIFO cleanup execution.
* `for await` rewrites a for-each loop into a `while` loop driven by
  `moveNext()`/`getCurrent()` calls, enclosed in `try`/`finally` for automatic
  stream cleanup.
* `yield return` signals the compiler to emit a generator that produces an
  `AsyncStream<T>` instead of an `Awaitable<T>`.

These transformations alter control flow, variable scoping, and exception-handling
boundaries â€” operations that a method call, which executes _within_ the existing
control flow, fundamentally cannot perform.

**Return-type rewriting.**
When the compiler encounters an `async` method returning `String`, it automatically
rewrites the return type to `Awaitable<String>` during the canonicalization phase â€”
_before_ type checking runs. If the method body contains `yield return`, the return
type becomes `AsyncStream<String>` instead. Because the keyword triggers this
rewriting automatically, the programmer declares the natural return type (`String`)
while the compiler handles the wrapping â€” keeping method signatures clean and
allowing the transformed type to flow through Groovy's type checker and IDE tooling
seamlessly.

**Guaranteed identity.**
Reserved keywords cannot be shadowed by local variables, overridden in subclasses,
or replaced by imported methods. This guarantee ensures that `await` carries the
same semantics in every source file, every class, and every scope. A global method,
by contrast, can be redefined â€” accidentally or intentionally â€” breaking async
behavior in ways that are difficult to diagnose.

[[yield-return]]
== `yield return` â€” Async Generators

The `yield return` statement enables async methods to produce a stream of values lazily.
A method containing `yield return` returns an `AsyncStream<T>` instead of a regular `Awaitable<T>`.
This is analogous to C#'s `yield return` within `IAsyncEnumerable<T>` and JavaScript's `async function*`
generators.

The producer (generator) and consumer (`for await` loop) coordinate via a
handoff queue, providing natural back-pressure: the producer blocks on each `yield return`
until the consumer is ready for the next value. This design means:

* The generator body runs on its own thread, executing sequentially like any normal method
* No values are buffered â€” the producer suspends until the consumer pulls the next item
* If the consumer stops early (`break`, `return`, exception), the producer thread is
interrupted, preventing resource leaks
* The generator can mix `yield return` with `await`, performing I/O between yields

NOTE: `yield return` can only appear inside an `async` method or `async` closure/lambda.
Using it elsewhere produces a compile-time error:
_"`yield return` can only be used inside an async method or async closure/lambda"_.

=== `first` â€” First Successful Result

`Awaitable.first(...)` returns the result of the first source that completes _successfully_,

* **Graceful degradation** â€” try a primary data source, then a fallback, then a cache
* **Redundant queries** â€” query multiple database replicas for improved tail latency

=== Choosing a Coordination Helper

Use the combinator that matches the lifecycle you want to enforce:

[cols="1,3,2"]
|===
| If you need... | Use... | Why

| every dependency must succeed before you continue
| `Awaitable.all(...)`
| It models "all-or-nothing" coordination and fails as soon as one dependency makes the overall result unusable.

| the first completion wins (success or failure)
| `Awaitable.any(...)`
| It expresses races, replicas, and "take the earliest answer" workflows where even errors are informative.

| the first _successful_ result, ignoring individual failures
| `Awaitable.first(...)`
| It is ideal for hedged requests, graceful degradation, and redundant queries where only success matters.

| the full outcome of every task, including failures
| `Awaitable.allSettled(...)`
| It is the right choice for reporting, cleanup, bulk processing, and partial-success workflows where you need every result.
|===

[[await-result]]
=== `AwaitResult` â€” Outcome Wrapper

Each element returned by `allSettled` is an `AwaitResult<T>`. This immutable value type
carries either a success value or a failure throwable. It follows the
_value object_ pattern (like Kotlin's `Result`, Rust's `Result<T, E>`),
providing proper `equals()` / `hashCode()` semantics and functional composition:

* `isSuccess()` / `isFailure()` â€” check outcome type
* `value` â€” the result (throws `IllegalStateException` on failure)
* `error` â€” the exception (throws `IllegalStateException` on success)
* `getOrElse(Function)` â€” recover from failures with a fallback function
* `map(Function)` â€” transform a successful value, propagating failures unchanged
* `equals(Object)` / `hashCode()` â€” two instances are equal if they carry the same state and equal values/errors
* `toString()` â€” returns `AwaitResult.Success[value]` or `AwaitResult.Failure[error]`

The `map` method enables functional composition chains without explicit unwrapping:

[source,groovy]
----
def result = AwaitResult.success("Groovy")
def length = result.map { it.length() }
assert length.value == 6

// map on a failure is a no-op â€” the error passes through
def failed = AwaitResult.failure(new IOException("err"))
def mapped = failed.map { it.toString() }
assert mapped.isFailure()
assert mapped.error.message == "err"
----

Value equality means `AwaitResult` instances work naturally in collections and assertions:

[source,groovy]
----
assert AwaitResult.success(42) == AwaitResult.success(42)
assert AwaitResult.success(1) != AwaitResult.success(2)

def set = [AwaitResult.success('a'), AwaitResult.success('a')] as Set
assert set.size() == 1  // deduplicated by equals/hashCode

[[flow-publisher]]
== `Flow.Publisher` Integration

Groovy's await system has built-in support for
{jdk}/java.base/java/util/concurrent/Flow.Publisher.html[`java.util.concurrent.Flow.Publisher`],
the reactive streams interface standardized in JDK 9. This enables seamless consumption of
reactive streams from any library that implements the `Flow.Publisher` contract (Project Reactor,
RxJava 3 via adapters, Vert.x, etc.).

The internal adapter uses a small bounded buffer to bridge the push-based `Publisher`
model to Groovy's pull-based `AsyncStream` model, preventing unbounded memory growth with
fast publishers while maintaining throughput.  Back-pressure is enforced by requesting exactly
one item at a time; demand for the next element is signalled _before_ the consumer's
`moveNext()` awaitable completes, so the publisher can begin producing the next value while
the consumer processes the current one.

The examples below use deterministic subscriber-handshake waiting with a bounded timeout
before submitting values, avoiding timing races and unbounded waits in CI environments.

Groovy ships dedicated adapter modules for the two most popular JVM reactive frameworks.
Adding either module to the classpath enables zero-configuration `async`/`await` support

The root async guide stays framework-neutral. Choose an integration module when your

[cols="1,1,1,2"]
|===
| Module | Single-value types | Streaming types | Typical fit

| Core runtime only
| `CompletableFuture`, `CompletionStage`, `Future`, `Flow.Publisher`

==== Project Reactor (`groovy-reactor`)

Use `groovy-reactor` when your public APIs, repositories, or HTTP boundaries already
expose `Mono` or `Flux`. The module lets Groovy async code consume those types directly
without rewriting the surrounding application model.

For verified examples covering operators, Spring WebFlux integration, and `Awaitable`
combinators, see the <<reactor-userguide,Project Reactor Integration>> guide.

==== RxJava 3 (`groovy-rxjava`)

Use `groovy-rxjava` when your application already standardizes on RxJava 3 and you want
to introduce Groovy async/await at orchestration boundaries without replacing `Single`,
`Maybe`, `Observable`, or `Flowable` in the rest of the codebase.

For verified examples covering `Maybe`, `Flowable`, back-pressure, and `Awaitable`
combinators, see the <<rxjava-userguide,RxJava 3 Integration>> guide.

==== Custom Adapters for Other Frameworks

For frameworks not covered by the built-in modules, implement the `AwaitableAdapter` interface
and delegate to the framework's native blocking or conversion methods:

* **Spring**: Return `Awaitable` from `@Async` service methods; adapt `ListenableFuture` via `toCompletableFuture()`
* **Vert.x**: Adapt `io.vertx.core.Future` via `toCompletionStage()`

=== Automatic Discovery

Adapters are discovered automatically via `java.util.ServiceLoader` by placing a
provider configuration file at `META-INF/services/groovy.concurrent.AwaitableAdapter`.
The `groovy-reactor` and `groovy-rxjava` modules use this mechanism â€” adding them to the
classpath is sufficient for transparent integration.

SPI-discovered adapters are loaded at class-initialization time and take priority over
the built-in adapters. Runtime-registered adapters (via `register()`) take highest
priority, allowing application code to override any default.

[[pitfall-scope-cancellation]]
=== Scope Cancellation and Uncooperative Tasks

Scope cancellation is cooperative. Child tasks should be written so they can reach a

[[async-modifier-restrictions]]
== `async` Modifier Restrictions

The `async` keyword is valid only on **method declarations** (including script-level methods)
and **closure/lambda expressions**. Applying `async` to other declarations is a compile-time error:

* **Class/interface declarations**: `async class Foo {}` â†’ parser error
* **Field/variable declarations**: `async def x = 1` â†’ compilation error ("async not allowed for variable declarations")
* **Constructor declarations**: `async Foo() {}` â†’ compilation error ("async not allowed for constructors")

[[implementation-details]]
== Design Notes

This section captures the stable design ideas behind Groovy's async/await support.
It intentionally focuses on semantics and responsibilities rather than internal helper
classes, so it remains useful as the implementation evolves.

[[implementation-strategy]]
=== Execution Model

Groovy preserves the ordinary control-flow model of the language: `async` methods still
look and behave like Groovy methods, `await` reads like an expression, and `try`/`catch`,
loops, and local variables keep their familiar meaning.

The runtime executes async bodies on JVM threads. On JDK 21+, this maps naturally to
virtual threads; on JDK 17â€“20, it maps to a managed executor. The key
user-visible consequences are stable across both strategies:

* stack traces and debugging stay close to ordinary Groovy code;
* blocking at an `await` boundary is handled by the runtime, not by callback plumbing;
* existing JVM async types can be adapted without forcing user code into one framework.

[[architecture]]
=== Compiler and Runtime Responsibilities

[cols="1,3"]
|===
| Layer | Responsibility

| Parser and compiler
| Recognize `async`, `await`, `defer`, `for await`, and `yield return`; validate where
they are legal; and rewrite declarations and control flow into the async runtime model.

| Runtime
| Schedule async bodies, coordinate waits and timeouts, bridge supported async types,
manage stream cleanup, and preserve context and scope across async boundaries.
|===

[[thread-safety]]
=== Behavioural Guarantees

[cols="2,3"]
|===
| Area | Guarantee

| Exception propagation
| `await` unwraps wrapper exceptions and rethrows the original failure type whenever possible.

| Stream lifecycle
| `for await` consumes values at the pace of the consumer and closes the source automatically
when the loop exits early.

| Structured concurrency
| Child tasks launched inside an `AsyncScope` are joined or cancelled before the scope exits.

| Context propagation
| Tasks and continuations see the async context that was active when they were created or registered.

| Interoperability
| External async frameworks integrate through public abstractions such as `Awaitable`,
`AsyncStream`, and the adapter registry rather than leaking framework-specific mechanics into user code.
|===

These guarantees are the part of the design application code should rely on. The exact
runtime strategy behind them may evolve, but the user-facing model is intended to remain stable.

[[structured-concurrency]]
== Structured Concurrency with `AsyncScope`

=== The Problem: Uncontrolled Task Lifetimes

When you launch async tasks with `async { ... }`, each task runs independently â€” it can
outlive the method that created it, leak resources if it is never awaited, or silently
swallow exceptions if nobody observes the result.  This "fire-and-forget" model is
convenient for simple cases but becomes a liability in production systems where every task
must be accounted for.

=== The Solution: Scoped Task Ownership

`AsyncScope` is an interface in `groovy.concurrent` that binds the lifetime of child
tasks to a well-defined scope.  Inspired by
Kotlin's `coroutineScope`, Swift's `TaskGroup`, and
{jdk}/java.base/java/util/concurrent/StructuredTaskScope.html[JEP 453 (Structured Concurrency)],
it enforces a simple invariant: **when the scope exits, all child tasks have completed or
been cancelled**.

Create a scope through the interface factories â€” `AsyncScope.create()`,
`AsyncScope.create(executor)`, or `AsyncScope.create(executor, failFast)` â€” or use the
recommended `AsyncScope.withScope { ... }` convenience form. This interface-based design
keeps application code tied to the public contract rather than any particular runtime class.

`AsyncScope` integrates naturally with `async` methods and `await` expressions.  You
can call `async` methods inside `scope.async { ... }` blocks and `await` their results â€”
just as you would in any other async context.  The `withScope` convenience method creates
a scope, executes a closure within it, and automatically closes the scope when the closure

By default, `AsyncScope` uses a _fail-fast_ policy: when any child task throws an

=== Thread Safety and Memory Efficiency

The `AsyncScope` contract is safe for concurrent use from multiple threads. In particular:

* **`async()` + `close()` cannot race.**  Child tasks are registered in the
  internal child list _before_ being submitted to the executor (the
  "register-before-submit" protocol).  This guarantees that `close()` always
  observes â€” and therefore joins or cancels â€” every submitted child, even
  under aggressive concurrent scheduling.  Once `close()` marks the scope as
  closed, no further children can be registered.  Any concurrent `async()` call
  that arrives after the flag is set receives an `IllegalStateException`.
* **`cancelAll()` is idempotent** and safe to call concurrently with `close()`.

Implementations are free to discard bookkeeping for completed children once it is no
longer needed, keeping long-lived scopes practical even when they spawn many short-lived tasks.
The runtime also propagates the current scope into child tasks automatically, so nested
async work inherits the correct structured-concurrency boundary.

[[go-tasks-and-channels]]
== Go-Style Tasks and Channels

Groovy's core async model is based on `Awaitable`, but it also provides
Go-inspired coordination primitives for workflows that benefit from explicit
message passing.

=== Spawning Tasks with `go`

`Awaitable.go { ... }` starts a lightweight task.  Outside an `AsyncScope` it
behaves like a detached async job.  Inside an `AsyncScope`, it automatically
joins the current scope so the task is awaited and cancelled with its siblings.

=== Channels

`AsyncChannel<T>` is a CSP (Communicating Sequential Processes) style interface
for inter-task communication.  `AsyncChannel.create()` creates an unbuffered
rendezvous channel; `AsyncChannel.create(n)` creates a buffered channel with
capacity `n`.  Both `send()` and `receive()` return `Awaitable` values, which
means they compose naturally with `await` and with `Awaitable.any(...)` for
first-completion-wins logic.

Key channel properties:

* **Unbuffered (capacity 0)**: `send()` suspends until a matching `receive()`
  arrives, providing direct handoff rendezvous semantics.
* **Buffered (capacity > 0)**: `send()` completes immediately if buffer space is
  available; otherwise it suspends until a consumer drains an element.
* **FIFO ordering**: values are always delivered in the order they were sent.
* **Close semantics**: closing a channel is idempotent.  Buffered values remain
  receivable after close.  Once drained, further receives fail with
  `ChannelClosedException`.  Pending senders fail immediately on close.
* **Null rejection**: channels reject `null` payloads to maintain a clean
  distinction between "no value yet" and actual data.
* **Thread safety**: all operations are safe for concurrent use by multiple
  threads.

[[performance-notes]]
== Performance Characteristics

=== Adapter Registry: Amortized O(1) Lookups

Every `await` expression resolves the awaited object's type to a matching
`AwaitableAdapter` via the `AwaitableAdapterRegistry`.  The registry uses
{jdk}/java.base/java/lang/ClassValue.html[`ClassValue`]-based per-class memoization,
reducing adapter resolution from O(n) (linear scan of registered adapters) to amortized
O(1) after the first lookup for each class.  The cache is automatically invalidated when
adapters are registered or unregistered at runtime.

=== Synchronous Completion Fast-Path

When an `Awaitable` wraps a `CompletableFuture` that has already completed by the time
`get()` is called, the runtime extracts the result via `CompletableFuture.join()` instead
of the full interruptible `get()` path.  This bypasses the park/unpark machinery and
eliminates unnecessary thread state transitions â€” a measurable win when async operations
complete before being awaited, which is common for cached or pre-computed results.

This optimisation is analogous to C#'s
https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.valuetask-1[`ValueTask<T>`]
synchronous completion path, where already-completed tasks skip the state machine
continuation entirely.

=== Virtual Threads (JDK 21+)

On JDK 21+, Groovy automatically uses
{jdk}/java.base/java/lang/Thread.html#ofVirtual()[virtual threads] for async execution.
Virtual threads are extremely lightweight â€” a few hundred bytes of stack compared to
~1 MB for platform threads â€” and are scheduled by the JVM's M:N scheduler onto a small
pool of carrier threads.  The `await` keyword blocks the virtual thread (not the carrier),
so the JVM can multiplex hundreds of thousands of concurrent async operations onto a
handful of OS threads.

On JDK 17â€“20, the runtime falls back to a bounded cached thread pool (default
256 threads, configurable via the `groovy.async.parallelism` system property) with a
caller-runs back-pressure policy.

[[design-rationale]]
== Design Rationale: Cross-Language Perspective

Groovy's async/await implementation sits at a unique point in the design space of mainstream
language concurrency primitives.  This section explains the key architectural choices and how
they compare to other languages.

=== Thread-Per-Task vs. State Machine

Most languages that implement async/await â€” C#, Kotlin, Rust, JavaScript â€” use a
*CPS (Continuation-Passing Style) transformation* or a *compiler-generated state machine*.
Each `await` point becomes a state transition, and the runtime resumes the method at the
correct point when the awaited operation completes.  This approach uses minimal memory per
suspended task (~200 bytes for a state object), but significantly increases compiler
complexity and can make debugging harder (stack traces show state machine internals).

Groovy instead uses a *thread-per-task model*: each `async` block runs on its own thread,
and `await` simply blocks that thread.  On JDK 21+, these are virtual threads with
comparable memory efficiency to state machines (~200 bytes per suspended virtual thread).
On older JDKs, real platform threads are used (bounded by the configurable pool size).

This choice has important implications:

[cols="1,2,2"]
|===
| Dimension | State Machine (C#, Kotlin, Rust) | Thread-Per-Task (Groovy)

| Compiler complexity
| High â€” CPS transform, state class generation
| Low â€” straightforward AST rewriting

| Debugger experience
| Synthetic stack frames; requires IDE support
| Natural thread stacks; works with all debuggers

| Blocking interop
| Dangerous â€” blocks the scheduler's thread pool
| Safe â€” blocking is the native execution model

| Memory per task (JDK 21+)
| ~200 bytes (state object)
| ~200 bytes (virtual thread)

| Memory per task (JDK < 21)
| ~200 bytes (state object)
| ~1 MB (platform thread stack)
|===

=== Exception Transparency

Groovy provides *checked exception transparency* through `deepUnwrap()` (recursive
unwrapping of `CompletionException` / `ExecutionException` layers up to depth 64) and
`sneakyThrow()` (type erasure trick to throw checked exceptions without declaring them).
This means `await` always rethrows the _original_ exception type, not a wrapper.

This is superior to C#'s `AggregateException` approach (which requires `.InnerException`
access) and JavaScript's untyped rejection model.  It is comparable to Kotlin's exception
transparency in coroutines.

=== Combinator Design

Groovy's combinator suite maps directly to JavaScript's Promise API:

[cols="2,2,3"]
|===
| Groovy | JavaScript | Behavior

| `Awaitable.all(a, b, c)`
| `Promise.all([a, b, c])`
| Wait for all; fail fast on first error

| `Awaitable.any(a, b)`
| `Promise.race([a, b])`
| First to settle (success _or_ failure) wins

| `Awaitable.first(a, b, c)`
| `Promise.any([a, b, c])`
| First _successful_ result; only fails when all fail

| `Awaitable.allSettled(a, b)`
| `Promise.allSettled([a, b])`
| Wait for all; collect every outcome as `AwaitResult`
|===

NOTE: `Awaitable.any()` corresponds to `Promise.race()` (first to _complete_), while
`Awaitable.first()` corresponds to `Promise.any()` (first to _succeed_).  The naming
difference from JavaScript is intentional: "any" implies any kind of completion, while
"first" implies the first result you actually want to use.
