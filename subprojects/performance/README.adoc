= Performance

This subproject contains two sets of performance related tests, compiler tests
and benchmarks.

== Compiler Performance Tests

The compiler tests can be run using the following Gradle task:

    ./gradlew :perf:performanceTests

This will compile various source files using several past versions of Apache
Groovy in addition to the current source version.

== Benchmarks

JMH Benchmarks can be run using:

    ./gradlew :perf:jmh

Groovy currently defaults to InvokeDynamic generated classes, but to get classic callsite generation use the `indy` property:

    ./gradlew -Pindy=false :perf:jmh

Groovy and Java sources placed in `src/test` will also be available to the
benchmarks.

To run a single benchmark or a matched set of benchmarks, use the
`benchInclude` property:

    ./gradlew -PbenchInclude=CallsiteBench :perf:jmh

The `benchInclude` property will perform a partial match against package
names or class names. It is equivalent to `.\*${benchInclude}.*`.

To attach JMH profilers (e.g. the `gc` profiler, which reports
`gc.alloc.rate.norm` in bytes/op), use the `jmhProfilers` property
(comma-separated):

    ./gradlew -PbenchInclude=FatFreeLambda -PjmhProfilers=gc :perf:jmh

== Ad-hoc benchmarks

Benchmarks under the `org.apache.groovy.adhoc` package are exploratory,
idiom-vs-idiom comparisons (for example, the `FatFreeLambda` set comparing
closure / `rcurry` idioms against the "fat-free" `java.util.function`
overloads and `curryWith` helpers). Their headline metric is *allocation*,
not throughput-over-time.

They are deliberately excluded from the `jmh`, `jmh-classic` and `jmh-daily`
CI suites (whose `benchInclude` patterns only match `.bench.` and `.perf.`).
Instead, the `jmh-adhoc` workflow runs them weekly (and on demand) with the
`gc` profiler, publishing throughput and allocation to their own dashboard
area under `dev/bench/jmh-adhoc/` so the core regression buckets stay
untouched. Run them locally with:

    ./gradlew -PbenchInclude='\.adhoc\.' -PjmhProfilers=gc :perf:jmh
