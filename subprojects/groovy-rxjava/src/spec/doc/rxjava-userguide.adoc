= RxJava 3 Integration

The `groovy-rxjava` module provides seamless integration between
https://github.com/ReactiveX/RxJava[RxJava 3] and Groovy's native
`async`/`await` system. Adding this module to the classpath enables
`await` on `Single` and `Maybe`, and `for await` on `Observable` and
`Flowable` — no manual adapter registration or boilerplate is required.

[[rxjava-getting-started]]
== Getting Started

Add `groovy-rxjava` as a dependency alongside Groovy:

.Gradle
[source,groovy,subs="attributes"]
----
dependencies {
    implementation "org.apache.groovy:groovy-rxjava:{groovy-full-version}"
}
----

.Maven
[source,xml,subs="attributes"]
----
<dependency>
    <groupId>org.apache.groovy</groupId>
    <artifactId>groovy-rxjava</artifactId>
    <version>{groovy-full-version}</version>
</dependency>
----

[[rxjava-type-mapping]]
== Type Mapping

The adapter maps all four core RxJava types to Groovy's async abstractions:

[cols="1,1,3"]
|===
| RxJava Type | Groovy Abstraction | Description

| `Single<T>`
| `Awaitable<T>`
| Emits exactly one value. Converted via `Single.toCompletionStage()`.

| `Maybe<T>`
| `Awaitable<T>`
| Emits zero or one value. An empty `Maybe` resolves to `null`.

| `Observable<T>`
| `AsyncStream<T>`
| Multi-value stream without back-pressure. Iterated via `for await`.

| `Flowable<T>`
| `AsyncStream<T>`
| Multi-value stream with Reactive Streams back-pressure. Iterated via `for await`.
|===

=== Choosing the Boundary

RxJava remains the right abstraction when your API is already modeled in RxJava terms.
Groovy async/await is most helpful where orchestration becomes easier to express as
straight-line code.

[cols="1,3"]
|===
| If your code is... | Prefer...

| exposing or composing a native RxJava API
| keep `Single`, `Maybe`, `Observable`, or `Flowable` in the public surface and use RxJava operators

| coordinating several async steps in service or application logic
| `await` for `Single` / `Maybe`, and `for await` for `Observable` / `Flowable`

| migrating gradually toward Groovy-native async abstractions
| `Awaitable.from(...)` or `AsyncStream.from(...)` where you want to cross that boundary explicitly
|===

[[rxjava-observable-vs-flowable]]
=== Observable vs. Flowable

[cols="1,1,2"]
|===
| Feature | `Observable` | `Flowable`

| Back-pressure
| No (buffering only)
| Yes (Reactive Streams)

| Typical use case
| UI events, small collections
| I/O streams, large datasets

| `for await` behaviour
| Buffers internally; consumer iterates at its own pace
| Requests elements one at a time; true demand-driven consumption

| Upstream interface
| `ObservableSource<T>`
| `Publisher<T>` (Reactive Streams)
|===

Both types work transparently with `for await`. Choose `Flowable` when
the producer may emit faster than the consumer can process.

[[rxjava-internals]]
== Operational Notes

For everyday development, the important guarantees are:

* adding `groovy-rxjava` to the classpath is enough for discovery — no manual adapter registration is required;
* `await` preserves the shape of single-result RxJava types: `Single` resolves one value, `Maybe` resolves one value or `null`, and failures propagate as exceptions;
* `for await` keeps stream consumption readable while preserving the practical difference between `Observable` and `Flowable`;
* leaving a `for await` loop early closes the adapted stream, which prevents accidental background work from continuing unseen.

[TIP]
====
Choose `Flowable` when you need explicit back-pressure across component boundaries.
Choose `Observable` for finite or moderate-rate event streams where that protocol would
add more ceremony than value.
====
