= Project Reactor Integration

The `groovy-reactor` module provides seamless integration between
https://projectreactor.io/[Project Reactor] and Groovy's native
`async`/`await` system. Adding this module to the classpath enables
`await` on `Mono` and `for await` on `Flux` — no manual adapter
registration or boilerplate is required.

[[reactor-getting-started]]
== Getting Started

Add `groovy-reactor` as a dependency alongside Groovy:

.Gradle
[source,groovy,subs="attributes"]
----
dependencies {
    implementation "org.apache.groovy:groovy-reactor:{groovy-full-version}"
}
----

.Maven
[source,xml,subs="attributes"]
----
<dependency>
    <groupId>org.apache.groovy</groupId>
    <artifactId>groovy-reactor</artifactId>
    <version>{groovy-full-version}</version>
</dependency>
----

Once on the classpath, the `ReactorAwaitableAdapter` is auto-discovered
via `java.util.ServiceLoader` at class-initialization time. No `@Import`,
no `register()` call, no configuration file — it just works:

[[reactor-type-mapping]]
== Type Mapping

The adapter maps Reactor types to Groovy's async abstractions as follows:

[cols="1,1,3"]
|===
| Reactor Type | Groovy Abstraction | Description

| `Mono<T>`
| `Awaitable<T>`
| Single-value asynchronous result. An empty `Mono` resolves to `null`.

| `Flux<T>`
| `AsyncStream<T>`
| Multi-value asynchronous sequence for use with `for await` loops.
|===

=== Choosing the Boundary

Use Reactor where Reactor already expresses the boundary of your application well, and
use Groovy's async syntax where sequential orchestration is easier to read and maintain.

[cols="1,3"]
|===
| If your code is... | Prefer...

| exposing a Reactor-first API to Spring WebFlux, repositories, or other Reactor code
| keep `Mono` / `Flux` in the public signature and compose with Reactor operators

| orchestrating several async calls in service or application logic
| `await mono` or `for await (item in flux)` inside Groovy async methods so the control flow stays linear

| moving gradually from Reactor APIs into Groovy-native abstractions
| `Awaitable.from(mono)` or `AsyncStream.from(flux)` at the point where you want a Groovy-facing contract
|===

[[reactor-internals]]
== Operational Notes

For day-to-day application code, the important guarantees are:

* adding `groovy-reactor` to the classpath is enough for discovery — no manual adapter registration is required;
* `await mono` preserves Reactor's single-result semantics: values resolve normally, empty monos resolve to `null`, and errors propagate as exceptions;
* `for await (item in flux)` lets the consumer drive the pace of iteration, which keeps streaming code easy to read and naturally bounded;
* leaving a `for await` loop early closes the adapted stream, so upstream work is not left running accidentally.

[TIP]
====
Stay in Reactor operators when you are still shaping a reactive pipeline. Switch to
`await` or `for await` at the point where imperative orchestration becomes clearer than
continued operator chaining.
====
