## Nature of changes

The migration was small and entirely concentrated in the build logic
(`build-logic/` convention plugins and a few `subprojects/*/build.gradle` files).
No application/source code under `src/` was touched.

### Transformation kinds applied (from `migration-data.json`)

- **`list`** — the dominant kind. Groovy DSL operator mutations (`+=`, `<<`) on
  `ListProperty` properties (`BaseForkOptions.jvmArgs`, `JavaForkOptions.jvmArgs`,
  `CompileOptions.compilerArgs`, `AntlrTask.arguments`, `JavaExecSpec.args`,
  `StandardJavadocDocletOptions.tags`) were rewritten to `.add(...)` / `.addAll([...])`
  / `.set([...])`, and one changed-return-type read (`args.asList()`) to `.get()`.
- **`dir`** — `AntlrTask.outputDirectory` is now a `DirectoryProperty`; a `.canonicalPath`
  read was resolved via `.get().asFile.canonicalPath`.
- **`file_collection`** — `MinimalJavadocOptions.classpath` is now a
  `ConfigurableFileCollection`; the `+=` append was rewritten to `.from(...)`.

### Manual fixes beyond the migration data

Three fixes were driven by build errors (tasks 07/08) rather than a direct
scanner hit, because the call sites used property-access / DSL forms the scanner's
`getX()` pattern does not match:

1. **`PerformanceTestsExtension.groovy` (task 07, `help`)** — `je.args.asList()` failed
   static type checking under `@CompileStatic` because `args` now returns
   `ListProperty<String>`. Resolved with `.get()`.
2. **`org.apache.groovy-core.gradle` (task 08, `assemble`)** — `generateGrammarSource`
   (AntlrTask) read `outputDirectory.canonicalPath` in a `doLast`; `outputDirectory` is
   now a `DirectoryProperty`. Resolved with `.get().asFile`.
3. **`org.apache.groovy-documented.gradle` (task 08, `assemble`)** — two issues in the
   shared `Javadoc` convention: `options.classpath += project.file(...)` hit the new
   `ConfigurableFileCollection` compound-assignment standin (which only accepts a
   `FileCollection`), rewritten to `.from(...)`; and three `addStringOption('tag', String)`
   calls failed because `'tag'` is now a *known multi-value* option backed by the
   `tags` `ListProperty`, rewritten to `tags.set([...])`.

### Known limitations / issues

- The build still emits the generic "Deprecated Gradle features were used … incompatible
  with Gradle 10" warning. Per the migration guidelines, deprecations were intentionally
  left for later; only removals/build-breaking changes were addressed.
- Verification covered `help` and `assemble`. Tests (`./gradlew test`) and the full
  `build` were not run as part of this workflow.
- The original `org.apache.groovy-documented.gradle` declared three custom Javadoc tags
  via repeated `addStringOption('tag', …)`. On Gradle 9.5.1 those calls were stored in a
  map keyed by option name, so only the last (`implNote`) effectively reached javadoc.
  The migrated `tags.set([...])` now passes all three tags — this is the documented
  intent and the only form the new known multi-value `'tag'` option accepts, but it is a
  minor behavioral correction rather than a byte-for-byte refactor.
